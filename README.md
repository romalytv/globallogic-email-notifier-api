# GitHub Release Notifier API 

REST API сервіс для підписки на email-сповіщення про нові релізи в репозиторіях GitHub. Написаний на чистому Node.js (Express) без використання важких фреймворків чи ORM, згідно з принципами "тонкої" архітектури.

## Технологічний стек
* **Backend:** Node.js, Express.js
* **Database:** PostgreSQL (чистий драйвер `pg` + `pg-pool`)
* **Jobs:** `node-cron`
* **Mailing:** `nodemailer`
* **Docs:** Swagger (`swagger-ui-express`, `yamljs`)
* **Infrastructure:** Docker, Docker Compose

---

## Встановлення та Запуск

### 1. Налаштування оточення
Створіть файл `.env` у корені проєкту. Оскільки налаштування підключення до бази даних вже прописані у `docker-compose.yml`, вам достатньо додати лише конфігурацію сервера, пошти та GitHub:

```env
PORT=3000

# GitHub API Token (можна вставити свій токен і збільшити ліміт запитів)
# GITHUB_TOKEN=gh_token_here

# Налаштування SMTP (Google App Passwords)
SMTP_HOST=smtp.gmail.com
SMTP_PORT=587
SMTP_USER=your_email_here
SMTP_PASS=your_app_password_here
```

### 2. Запуск через Docker (Рекомендовано)
У проєкті налаштовано багатоетапну збірку та оркестрацію бази даних. Виконайте одну команду:

```bash
docker-compose up --build
```

Сервіс автоматично підніме базу даних PostgreSQL, виконає міграції таблиць, запустить API на порту 3000 та увімкне фоновий сканнер релізів.

### 3. Запуск без Docker (Локально)
Якщо ви хочете запустити проєкт локально через `Node.js`, вам необхідно додати змінні бази даних у ваш .env файл. За замовчуванням :
```
DB_HOST=localhost
DB_USER=postgres
DB_PASSWORD=secret
DB_NAME=github_notifier
```
та підняти PostgreSQL власноруч. Потім виконати:

```bash
npm install
npm start
```

## Архітектура та Логіка

* **База даних:** Міграції таблиць виконуються автоматично при старті сервера. Цілісність даних гарантується на рівні БД (`ON DELETE CASCADE`, композитні `UNIQUE` індекси).
* **Оптимізований Сканнер:** `node-cron` перевіряє GitHub лише для тих репозиторіїв, які мають активних підписників. Код коректно обробляє ліміти GitHub API (`429/403`), безпечно перериваючи цикл.
* **Безпека та Сповіщення:** Для підтвердження емейлу та відписки використовуються унікальні 64-символьні hex-токени. Усі листи містять кнопку `Unsubscribe`.
* **Сумісність API:** 100% відповідність Swagger-документації завдяки DTO-мапінгу результатів з бази. Сервіс автоматично адаптується до роботи як із GitHub-токеном, так і без нього.

# ЕКСТРА

## API key автентифікація: ендпоїнти захищені токеном у заголовку.

Для демонстрації API KEY можливостей у проєкт було впроваджено шар авторизації за допомогою заголовка X-API-KEY.

Особливості реалізації:
Стратегія "Soft Auth": Щоб забезпечити 100% сумісність із наданим контрактом swagger.yaml та не порушувати роботу автоматичних тестів, авторизація є опціональною.

Логіка роботи: Сервер пропускає запити без заголовка (наприклад, зі Swagger UI), але якщо заголовок X-API-KEY присутній у запиті — він проходить сувору перевірку на відповідність секретному ключу сервера.

Інтеграція з фронтендом: Веб-інтерфейс проєкту повністю налаштований на роботу з цим заголовком.

**ВАЖЛИВО!!** - Для того щоб перевірити що токен працює ліпше всього підкорректувати константу `const API_KEY` за шляхом:
```
/public/index.js
```

**ВАЖЛИВО!!** - Минулу версію релізу довелося знести та піднімати нову через вказання GitHub щодо висвітлення API ключа в `/public/index.js`, то ж сам ключ був спрощений. (У разі незрозумілостей прошу відкрити нову Issue.)
